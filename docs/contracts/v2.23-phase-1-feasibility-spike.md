# ShadowBid V2.23 Phase 1 Feasibility Spike

## Purpose

This document defines the first execution phase for `v2.23`.

The point of Phase 1 is not to ship the whole contract. It is to answer the hardest technical questions before the team freezes a new auction ABI or promises full privacy.

## Current Decision Snapshot

The `USDCx` private-first path is currently paused for `v2.23`.

Current planning position:

- keep the Phase 1 artifacts and sidecar work as reference material
- do not freeze the `v2.23` ABI
- treat self-built `get_credentials` proof generation from public mappings as a current no-go
- only reopen this path if a wallet, vendor, issuer, or off-chain proof builder can supply the missing proof material or a usable `Credentials` record

## Phase 1 Goal

Prove that ShadowBid can support a `USDCx` private-first bid funding path that:

- does not leak the bid amount through a public transfer path
- still supports cryptographic commit/reveal verification
- is operationally realistic in wallet flow, proving time, and setup overhead

## Phase 1 Non-Goals

Phase 1 should not try to complete all of `v2.23`.

Out of scope:

- full multi-currency privacy parity
- production-ready batch settlement
- delegated reveal redesign
- final UI polish
- final deployment cutover

## Required Deliverables

Phase 1 should produce all of the following:

1. a private `USDCx` funding spike
2. a benchmark harness for ShadowBid-shaped workloads
3. a written escrow custody decision record
4. a go / no-go decision on freezing the `v2.23` auction ABI

If any of these are missing, Phase 1 is incomplete.

## Work Items

### Step 1: Define The Private Funding Experiment

Questions to answer:

- can a bidder commit to an auction using private `USDCx` records plus credentials without exposing amount publicly?
- what exact inputs are required from wallet, SDK, and token program?
- can the flow fit inside a realistic user journey?

Expected artifact:

- one written experiment spec inside `scripts/v223/README.md`

### Step 2: Build A Minimal Spike

The minimum spike should prove one private bid path end to end:

- prepare private `USDCx` funding inputs
- generate commitment preimage
- submit the private funding + commit path
- verify no public per-bid amount leak occurs in the expected places

Expected artifact:

- a reproducible script or workflow under `scripts/v223/`

### Step 3: Benchmark ShadowBid-Shaped Workloads

Do not rely only on generic batch-payment benchmarks.

Run at least these scenarios:

- single private commit
- single private refund
- single winner settlement path
- two auctions settling in parallel
- one refund-heavy scenario

For each scenario, capture:

- proving time
- wall-clock time
- fee cost
- number of records required
- number of credentials required
- number of user actions required

Expected artifact:

- machine-readable result files under `contracts/benchmarks/v223-private-usdcx/`

### Step 4: Write The Escrow Custody Decision

This is the most important design output of Phase 1.

The team must write down:

- where the asset lives after private funding
- who can later move it
- how refunds are authorized
- how seller payout is authorized
- how platform fee payout is authorized
- what stays public and what stays private

Expected artifact:

- `docs/contracts/v2.23-custody-decision.md`

### Step 5: Produce A Go / No-Go Report

The phase should end with a short decision report.

It must answer:

- is the private funding path technically viable?
- is the custody model clear enough to freeze an ABI?
- are proving times and fees acceptable for a user-facing auction flow?
- should `v2.23` continue as `USDCx`-private-first?

Expected artifact:

- `docs/contracts/v2.23-benchmark-findings.md`

## Benchmark Matrix

The Phase 1 benchmark matrix should be intentionally small.

### Required Scenarios

| Scenario | Why it matters |
| --- | --- |
| private commit, 1 bidder | proves the basic funding rail |
| private commit + reveal, 1 bidder | proves commitment compatibility |
| private commit + cancelled refund | proves loser/cancel path viability |
| private commit + winner settlement | proves the happy path viability |
| 2 auctions in parallel | exposes operational bottlenecks |

### Required Metrics

| Metric | Reason |
| --- | --- |
| proving time | user wait and executor cost |
| wall-clock time | end-to-end latency |
| fee total | economic viability |
| records consumed | setup pressure |
| credentials consumed | compliance overhead |
| public leak check | core privacy requirement |

## Exit Criteria

Phase 1 is successful only if all of the following are true:

- a private `USDCx` bid funding path is demonstrated
- the amount does not leak through the tested public funding path
- the commitment still verifies against the reveal preimage
- at least one refund-oriented path is demonstrated
- at least one winner-oriented path is demonstrated
- the team writes the custody decision down clearly
- the benchmark findings support either:
  - continuing with `v2.23`, or
  - stopping before the ABI is frozen

## No-Go Signals

The team should stop and redesign before freezing the ABI if any of the following happen:

- the private funding path still leaks the amount publicly
- the required `get_credentials` proof payload cannot be sourced without undocumented external capability
- the custody model depends on undefined or hand-wavy off-chain trust
- wallet UX requires too many brittle manual steps
- proving time or fee cost is too high for realistic auction usage
- refund or payout semantics become much weaker than `v2.22`
