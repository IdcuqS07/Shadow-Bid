# ShadowBid V2.23 Escrow Custody Decision

## Status

Draft. This document is a blocking design gate for `v2.23`.

Do not freeze the `v2.23` auction ABI until the decision recorded here is stable and backed by benchmark evidence.

## Purpose

`v2.22` fixed the public-escrow mapping flaw and the non-cryptographic commitment flaw, but it still leaks bid amounts through public funding paths.

`v2.23` must solve more than commitment storage. It must define where privately funded assets actually live between commit and settlement, and who can later move them without quietly re-introducing off-chain trust.

## Decision Summary

The current project decision is:

1. `v2.23` may continue only if it proves a private funding path and a custody model that preserves locked-funds semantics.
2. `bidder self-custody after commit` is rejected for the final auction design because it does not actually lock the bid amount.
3. `operator omnibus custody` is rejected for the final auction design because it makes auction safety depend on undefined off-chain trust.
4. `contract-verifiable private custody` remains the target architecture for a real `v2.23` private release.
5. If Phase 1 cannot show a credible path toward contract-verifiable private custody, the team should stop before freezing the ABI and keep `v2.22` as the honest live version.

## Why This Is A Blocking Issue

ShadowBid is not only trying to hide bid amounts. It also needs to preserve the auction guarantees that make the marketplace meaningful:

- funded bids should be locked for the auction lifecycle
- loser refunds should be deterministic
- winner settlement should match the revealed and validated outcome
- seller payout and platform fee payout should not depend on hand-wavy operator behavior

If the custody model is weak, the system may be more private on paper while being less trustworthy in practice.

## Evaluated Custody Models

### Option A: Bidder Self-Custody After Commit

Description:

- the bidder keeps the private `USDCx` record after commit
- the contract stores only commitment and auction metadata
- the bidder is expected to pay later if they win

Result:

- rejected for the final private auction path

Why:

- funds are not actually locked
- a winning bidder can fail to cooperate later
- refund semantics become meaningless because there was never real escrow

### Option B: Operator-Held Omnibus Private Escrow

Description:

- bidders privately transfer `USDCx` into a wallet controlled by a platform or executor operator
- the contract stores the commitment and public lifecycle state
- later refunds and payouts depend on the operator moving the private records correctly

Result:

- rejected as the final `v2.23` architecture
- acceptable only as a narrow feasibility probe if it is explicitly labeled a temporary experiment

Why:

- it adds off-chain custody trust that the current contract model does not describe clearly
- private amount hiding improves, but trust assumptions degrade
- platform insolvency, censorship, or operational mistakes become harder to reason about

### Option C: Contract-Verifiable Private Escrow

Description:

- the private funding path moves assets into a custody model whose later refund and payout behavior is constrained by contract-verifiable auction outcomes
- the contract still stores the minimum public state needed for lifecycle progress
- later settlement actions remain bound to reveal, reserve, and dispute results

Result:

- accepted as the target architecture
- still unproven and must be demonstrated before ABI freeze

Why:

- this is the only direction that keeps both privacy and meaningful escrow semantics
- it matches the product claim the project actually wants to make
- it prevents `v2.23` from becoming a privacy patch that weakens auction guarantees

## Working Phase 1 Decision

For Phase 1, the team should split the work into two layers:

1. prove the private `USDCx` funding rail and verify that amount leakage is removed from the tested path
2. refuse to market or ABI-freeze `v2.23` until the custody model is shown to preserve locked-funds, refund, seller payout, and platform fee semantics

This means a successful funding spike is necessary, but not sufficient.

## Required Questions Before ABI Freeze

The team must write down clear answers to all of the following:

- where does the privately funded asset live after commit?
- what on-chain state binds that asset to one auction and one bidder?
- who is allowed to move the asset on refund?
- who is allowed to move the asset on seller payout?
- who is allowed to move the asset on platform fee payout?
- what exact evidence proves that a payout amount matches the validated auction result?
- which data remains public and which data remains private?

If any answer still depends on "the operator will do the right thing," the custody model is not ready.

## Benchmark Evidence Required

This decision should only move from draft to accepted when the benchmark artifacts show:

- one private funding path without public amount leakage
- one refund-oriented path tied to the same custody model
- one winner-oriented path tied to the same custody model
- operational notes on record count, credentials, and user actions
- explicit notes about remaining trust assumptions

## No-Go Rule

Do not ship `v2.23` as a hidden-amount sealed-bid release if the best available custody model still depends on informal off-chain trust.

In that case, keep the live product positioned as `v2.22` plus research toward private escrow.
