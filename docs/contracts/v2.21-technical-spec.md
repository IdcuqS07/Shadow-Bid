# ShadowBid v2.21 Technical Spec Draft

## Status

- Draft
- Derived from the live `v2.20` contract in `contracts/src/main.leo`
- Derived from the autonomy blueprint in `docs/contracts/autonomous-lifecycle-design.md`
- Intended to be implementation-ready enough for contract and frontend planning

## Scope

`v2.21` should focus on one concrete upgrade path:

1. split overloaded lifecycle deadlines
2. make the post-reveal path deterministic
3. prevent auctions from stalling when no bidder reveals
4. keep the existing state enum as stable as possible for the frontend

Out of scope for `v2.21`:

- delegated reveal authority
- encrypted reveal escrow
- threshold-key keeper infrastructure
- private-record-backed bid escrow
- cryptographic commitment replacement for `nonce + amount`

Those items remain valid follow-ups for a later release.

## Design Choice

This draft chooses the smallest contract change that solves the current operational gap.

Instead of adding a large new reveal-authority subsystem in `v2.21`, the contract should:

- keep the existing high-level state machine
- split `challenge_period` into separate reveal and dispute windows
- replace overloaded `challenge_end_time` with explicit deadlines
- add a deterministic timeout transition that either advances to `CHALLENGE` or cancels the auction

This keeps the migration surface smaller while still fixing the main failure mode from `v2.20`.

## State And Storage Changes

### State Enum

The recommended `v2.21` core state enum stays unchanged:

```text
OPEN       = 0
CLOSED     = 1
CHALLENGE  = 2
SETTLED    = 3
CANCELLED  = 4
DISPUTED   = 5
```

Reason:

- `v2.20` frontend code already understands these values
- the main issue is not missing states, but overloaded timing and missing timeout fallback
- avoiding new enum values reduces parser, UI, and validation churn

### `AuctionRecord` Changes

`AuctionRecord` should be updated so the seller receipt matches the new lifecycle model.

```text
AuctionRecordV21 {
  owner: address,
  auction_id: field,
  min_bid_amount: u128,
  currency_type: u8,
  end_time: i64,
  reveal_period: i64,
  dispute_period: i64
}
```

### `AuctionInfo` Changes

The key `v2.21` storage change is replacing overloaded timing fields.

```text
AuctionInfoV21 {
  seller: address,
  min_bid_amount: u128,
  currency_type: u8,
  end_time: i64,

  reveal_period: i64,
  dispute_period: i64,

  state: u8,
  winner: address,
  winning_amount: u128,

  reveal_deadline: i64,
  dispute_deadline: i64,

  total_escrowed: u128,
  asset_type: u8,
  item_received: bool,
  item_received_at: i64,
  payment_claimed: bool,
  payment_claimed_at: i64,
  confirmation_timeout: i64,
  settled_at: i64,
  claimable_at: i64,
  platform_fee_amount: u128,
  seller_net_amount: u128,
  platform_fee_claimed: bool,
  platform_fee_claimed_at: i64,
  reserve_price: u128,
  reserve_met: bool
}
```

### Field-Level Delta

| `v2.20` field | `v2.21` change | Notes |
| --- | --- | --- |
| `challenge_period` | replaced by `reveal_period` and `dispute_period` | one duration is no longer reused for two different windows |
| `challenge_end_time` | replaced by `reveal_deadline` and `dispute_deadline` | removes ambiguous semantics after close and winner determination |
| `state` | unchanged | compatibility-first decision |
| `reserve_met` | unchanged | still derived once the winning amount is known |
| `claimable_at` | unchanged | still derived from `settled_at + confirmation_timeout` |

### `BidCommitment` Changes

`v2.20` stores a `timestamp` field that is not currently informative. `v2.21` should make the reveal audit trail explicit.

```text
BidCommitmentV21 {
  bidder: address,
  commitment: field,
  is_revealed: bool,
  revealed_amount: u128,
  revealed_at: i64
}
```

### `Escrow` Changes

No schema change is required for the `v2.21` core fallback design.

```text
Escrow {
  bidder: address,
  is_refunded: bool,
  is_winner: bool
}
```

Source-patch note:

The remediated source removes `amount` from `Escrow`, so the public mapping no longer exposes per-bid amounts directly.

Important deployment note:

The active remediated testnet deployment now uses this no-amount `Escrow` shape in `shadowbid_marketplace_v2_22.aleo`. The older `shadowbid_marketplace_v2_21.aleo` program still uses the public-amount `Escrow` shape.

### Mapping Changes In Leo

The recommended `v2.21` core does not require new top-level mappings.

Existing mappings remain:

- `auctions`
- `commitments`
- `escrow`
- `highest_bid`
- `highest_bidder`
- `seller_profiles`
- `auction_proof_root`
- `auction_disclosure_root`
- `auction_dispute_root`

The important change is the shape of data stored inside:

- `mapping auctions: field => AuctionInfoV21`
- `mapping commitments: field => BidCommitmentV21`

### Explicitly Deferred Mappings

These were proposed in the autonomy design, but are intentionally deferred out of `v2.21` core:

- `reveal_delegate`
- `reveal_package_root`
- `reveal_delegate_status`
- `reveal_ciphertext_root`
- `reveal_escrow_mode`
- `reveal_escrow_status`

## Transition Changes

### 1. `create_auction`

Status:
- changed signature

Old input:

```text
(auction_id, min_bid_amount, reserve_price, currency_type, asset_type, end_time, challenge_period)
```

New input:

```text
(auction_id, min_bid_amount, reserve_price, currency_type, asset_type, end_time, reveal_period, dispute_period)
```

Required checks:

- `reveal_period > 0`
- `dispute_period > 0`
- `reserve_price >= min_bid_amount`
- existing currency and asset validation remains

Post-state:

- `state = OPEN`
- `reveal_deadline = 0`
- `dispute_deadline = 0`
- `winner = EMPTY_ALEO_ADDRESS`
- `winning_amount = 0`

### 2. `close_auction`

Status:
- behavior changed

Required checks:

- `state == OPEN`
- `closed_at >= end_time`

Post-state:

- `state = CLOSED`
- `reveal_deadline = closed_at + reveal_period`
- `dispute_deadline = 0`

Important behavior change:

- `close_auction` must no longer write to a generic `challenge_end_time`

### 3. `reveal_bid`

Status:
- behavior changed

Required checks:

- `state == CLOSED`
- `revealed_at <= reveal_deadline`
- commitment matches
- `!is_revealed`
- `amount >= min_bid_amount`
- `escrow.is_refunded == false`

Post-state:

- `commitment.is_revealed = true`
- `commitment.revealed_amount = amount`
- `commitment.revealed_at = revealed_at`
- `highest_bid` and `highest_bidder` update if this reveal beats the current leader

No state transition occurs here. The auction stays `CLOSED` until timeout settlement runs.

### 4. `settle_after_reveal_timeout`

Status:
- new transition

Purpose:

- deterministically resolve the reveal window after `reveal_deadline`
- replace the combination of `determine_winner` plus the no-reveal gap
- collapse reserve-miss fallback into the same timeout step

Recommended signature:

```text
async transition settle_after_reveal_timeout(
  public auction_id: field,
  public settled_at: i64
) -> Future
```

Required checks:

- `state == CLOSED`
- `settled_at >= reveal_deadline`

Resolution rules:

1. If no valid reveal exists:
   - set `state = CANCELLED`
   - keep `winner = EMPTY_ALEO_ADDRESS`
   - keep `winning_amount = 0`
   - set `reserve_met = false`

2. If a valid reveal exists but `highest_bid < reserve_price`:
   - set `state = CANCELLED`
   - set `winner = highest_bidder`
   - set `winning_amount = highest_bid`
   - set `reserve_met = false`

3. If a valid reveal exists and `highest_bid >= reserve_price`:
   - set `state = CHALLENGE`
   - set `winner = highest_bidder`
   - set `winning_amount = highest_bid`
   - set `reserve_met = true`
   - set `dispute_deadline = settled_at + dispute_period`

Notes:

- this transition should be permissionless, like the current keeper-friendly lifecycle
- it is safe because the result is fully derived from existing mapping state
- it makes the auction continue even if some or all bidders never return after commit

### 5. `determine_winner`

Status:
- deprecated in `v2.21`

Reason:

- `settle_after_reveal_timeout` replaces it with a more complete timeout resolution step
- the new transition handles:
  - at least one reveal
  - no reveals
  - reserve not met

### 6. `cancel_auction_reserve_not_met`

Status:
- deprecated in `v2.21`

Reason:

- reserve handling should be decided inside `settle_after_reveal_timeout`
- the seller should no longer need a separate cleanup action when reserve is missed

### 7. `finalize_winner`

Status:
- behavior changed

Required checks:

- `state == CHALLENGE`
- `finalized_at >= dispute_deadline`
- no active dispute root
- `reserve_met == true`

Post-state:

- `state = SETTLED`
- `settled_at = finalized_at`
- `claimable_at = finalized_at + confirmation_timeout`
- existing platform-fee and seller-net calculations remain

Important behavior change:

- `finalize_winner` must no longer read `challenge_end_time`
- it must read `dispute_deadline`

### 8. `open_dispute`

Status:
- mostly unchanged

Recommended checks remain:

- caller is seller or winner
- no existing dispute root
- payment not already claimed
- platform fee not already claimed

Recommended timing rule:

- allow opening while in `CHALLENGE`
- allow opening while in `SETTLED` before payout completion

This keeps current dispute behavior compatible with the live flow.

## Before/After Transition Rules

| Transition | Before | After |
| --- | --- | --- |
| `create_auction` | no existing auction | `OPEN` with zero deadlines |
| `close_auction` | `OPEN` and auction ended | `CLOSED` with `reveal_deadline` |
| `reveal_bid` | `CLOSED` and before `reveal_deadline` | still `CLOSED`, commitment marked revealed |
| `settle_after_reveal_timeout` | `CLOSED` and after `reveal_deadline` | `CHALLENGE` if valid winner and reserve met, otherwise `CANCELLED` |
| `finalize_winner` | `CHALLENGE` and after `dispute_deadline` | `SETTLED` |
| `open_dispute` | `CHALLENGE` or `SETTLED` before payout completion | `DISPUTED` |
| `resolve_dispute_release_seller` | `DISPUTED` | `SETTLED` |
| `resolve_dispute_refund_winner` | `DISPUTED` | `CANCELLED` |


## Edge Cases And Fallback Behavior

### 1. Auction closed with zero commits

Behavior:

- `settle_after_reveal_timeout` should still be valid after `reveal_deadline`
- result should be `CANCELLED`

### 2. Auction has commits but zero reveals

Behavior:

- `settle_after_reveal_timeout` cancels the auction
- all bidders later use the existing refund path for `CANCELLED`

### 3. Auction has reveals but reserve is not met

Behavior:

- `settle_after_reveal_timeout` cancels the auction directly
- `cancel_auction_reserve_not_met` is no longer needed

### 4. Late reveal after timeout settlement already ran

Behavior:

- must fail because auction state is no longer `CLOSED`

### 5. Invalid chain state where `highest_bid` exists but commitment reveal data is inconsistent

Behavior:

- transition should fail rather than trying to repair silently
- ops should inspect mappings before retrying

### 6. Dispute opened after timeout settlement

Behavior:

- still valid if the auction reached `CHALLENGE` or `SETTLED`
- not valid if the timeout path already cancelled the auction

## Migration Notes From `v2.20` To `v2.21`

### Contract Migration Strategy

There is no in-place storage migration between Aleo program IDs.

Recommended rollout:

1. deploy a new program ID, for example `shadowbid_marketplace_v2_21.aleo`
2. keep all existing `v2.20` auctions on the `v2.20` settlement path
3. create only new auctions on `v2.21` after frontend cutover

Reason:

- `AuctionInfo` field layout changes
- `AuctionRecord` field layout changes
- `BidCommitment` field layout changes
- existing mappings cannot be safely reinterpreted as `v2.21`

### Transition Migration Notes

| `v2.20` transition | `v2.21` status | Migration note |
| --- | --- | --- |
| `create_auction` | changed | pass `reveal_period` and `dispute_period` |
| `close_auction` | changed | now writes `reveal_deadline` |
| `reveal_bid` | changed | now checks `reveal_deadline` and writes `revealed_at` |
| `determine_winner` | deprecated | replace call sites with `settle_after_reveal_timeout` |
| `cancel_auction_reserve_not_met` | deprecated | reserve-miss handling moves into timeout settlement |
| `finalize_winner` | changed | now checks `dispute_deadline` |

### Frontend Migration Notes

These repo files are expected to change for `v2.21` support:

- `frontend/src/services/aleoServiceV2.js`
- `frontend/src/utils/auctionStateValidator.js`
- `frontend/src/lib/auctionUtils.js`
- `frontend/src/pages/PremiumAuctionDetail.jsx`

Required frontend updates:

1. change the default program ID from `shadowbid_marketplace_v2_20.aleo` to `shadowbid_marketplace_v2_21.aleo`
2. replace parsing of `challenge_period` with `reveal_period` and `dispute_period`
3. replace parsing of `challenge_end_time` with `reveal_deadline` and `dispute_deadline`
4. remove UI assumptions that winner selection always comes from `determine_winner`
5. add UI copy for the timeout resolution path

Important compatibility note:

- because the recommended state enum stays at `0..5`, most status rendering code can stay stable
- field parsing logic still must change

### Ops / Keeper Migration Notes

The keeper or ops layer should add one new automated branch:

1. if auction is `CLOSED` and current time is before `reveal_deadline`, wait
2. if auction is `CLOSED` and current time is after `reveal_deadline`, call `settle_after_reveal_timeout`
3. if auction becomes `CHALLENGE`, wait for `dispute_deadline`
4. if auction remains dispute-free after `dispute_deadline`, call `finalize_winner`

This is the minimum automation upgrade needed to benefit from `v2.21`.

### Local Storage And Pending-Action Notes

Local pending-action keys should include the program ID or contract version.

Reason:

- `v2.20` and `v2.21` will coexist during rollout
- deadline fields and next-action logic differ between versions
- mixed local cache entries could produce incorrect reminders or auto-actions

## Recommended Acceptance Criteria

`v2.21` is complete when the following scenarios pass:

1. auction with valid reveals and reserve met reaches `CHALLENGE` through `settle_after_reveal_timeout`
2. auction with valid reveals but reserve missed reaches `CANCELLED` through `settle_after_reveal_timeout`
3. auction with zero valid reveals reaches `CANCELLED` through `settle_after_reveal_timeout`
4. `finalize_winner` uses `dispute_deadline`, not a reused reveal deadline
5. losing bidders can still claim refunds after timeout-driven cancellation
6. existing dispute transitions still work for auctions that entered `CHALLENGE`

## Deferred `v2.22+` Work

After `v2.21` is stable, the next layer can add:

- delegated reveal authority
- reveal-package roots
- encrypted reveal escrow
- penalty-backed non-reveal economics if needed

That later work should build on the cleaner deadline model introduced in `v2.21`.
