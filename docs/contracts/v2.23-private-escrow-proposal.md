# ShadowBid V2.23 Private Escrow Proposal

## Goal

`v2.22` fixed the exact Wave 4 code-level flaws in ShadowBid, but it did not yet deliver fully hidden-amount sealed-bid privacy.

The purpose of `v2.23` should be narrower and more honest:

- remove amount leakage from the bid funding path, not just from public mappings
- preserve the working auction lifecycle from `v2.22`
- avoid over-promising privacy for currencies or flows that are not actually private yet

Current planning status:

- this proposal is now a design reference, not an active release plan
- the `USDCx` private-first path is paused until an external `get_credentials` proof source exists
- `v2.23` should not be treated as ready to freeze around this rail

## Current Starting Point

`v2.22` already does two important things:

- it stops storing per-bid amounts in the public `escrow` mapping
- it replaces the old arithmetic commitment with an in-contract `BHP256` commitment

What still leaks today is the funding path itself. Public `ALEO`, `USDCx`, and `USAD` transfers can still expose bid amounts before reveal.

## What The External Benchmark Repo Proves

Reference repository:

- <https://github.com/AleoAlexander/batch-payments-benchmarking>

The benchmark repo is not an auction system, but it does prove several patterns that are directly useful for ShadowBid.

### 1. Private USDCx Transfers Are Practical

The repo uses `test_usdcx_stablecoin.aleo/transfer_private_with_creds` as the main payment primitive for private USDCx sends:

- <https://github.com/AleoAlexander/batch-payments-benchmarking/blob/main/src/programs/batch_usdcx/src/main.leo>

That matters because it shows ShadowBid does not need to guess whether a private USDCx rail exists. It does.

### 2. Record Splitting Is A Useful Utility Layer

The repo includes a dedicated `split_usdcx` Leo program that turns one larger private token record into many smaller records:

- <https://github.com/AleoAlexander/batch-payments-benchmarking/blob/main/src/programs/split_usdcx/src/main.leo>

This is valuable for ShadowBid because refunds, seller payouts, and operational settlement often work better when the executor already has the right denomination and record count prepared ahead of time.

### 3. Credential Batching Can Reduce Setup Overhead

The repo includes a batched credential minting helper for USDCx:

- <https://github.com/AleoAlexander/batch-payments-benchmarking/blob/main/src/programs/batch_usdcx_credentials/src/main.leo>

This suggests ShadowBid should treat credential provisioning as a first-class systems problem if private USDCx becomes the main privacy rail.

### 4. Benchmark Harness Matters

The repo does not just ship programs. It also ships setup and runner logic for:

- record splitting
- credential provisioning
- concurrency experiments
- fee and wall-clock measurement

Useful references:

- <https://github.com/AleoAlexander/batch-payments-benchmarking/blob/main/src/shared/setup-strategy.ts>
- <https://github.com/AleoAlexander/batch-payments-benchmarking/blob/main/src/shared/run-strategy.ts>

ShadowBid should copy this habit even if it does not copy the exact code.

### 5. Aleo Limits Need To Shape The Design

The benchmark repo calls out constraints that matter for `v2.23` planning:

- max transitions per transaction is effectively limited
- async capture count limits practical batch size
- large batch programs can hit deployment variable limits
- batch proving cost rises quickly with batch density

The README explicitly notes these constraints:

- <https://github.com/AleoAlexander/batch-payments-benchmarking/blob/main/README.md>

## What The External Repo Does Not Solve

This is the most important caution.

The benchmark repo proves private payment and batching patterns. It does not prove a complete private sealed-bid auction design.

Specifically, it does not solve:

- auction state management
- commit/reveal winner selection
- dispute handling
- reserve-aware auction settlement
- long-lived private escrow custody inside the auction protocol

Inference from the source: the repo is designed for private payment execution, not for contract-enforced auction escrow. ShadowBid still needs its own custody and settlement model on top.

## Recommended V2.23 Scope

The safest and most useful `v2.23` scope is:

### 1. Make V2.23 USDCx-Private-First

This section now describes the reopening condition for the paused path, not an active commitment.

Do not try to make all three currencies "fully private" in one jump.

Recommended scope:

- private sealed-bid mode only for `USDCx`
- keep `ALEO` and `USAD` either on the public `v2.22` path or clearly labeled as non-private modes
- avoid any UI language implying equal privacy across all currencies

Why:

- the benchmark repo only demonstrates a serious private rail for `USDCx`
- USDCx already has credential-aware private transfer primitives
- this reduces design risk and prevents another over-broad privacy claim
- however, the path should only reopen if an external proof source can supply the missing `get_credentials` inputs or a usable `Credentials` record

### 2. Keep Batch Logic Out Of The Auction Core

Do not make the main auction contract depend on large array-based batch transitions for normal bidding.

Recommended split:

- core auction contract stays focused on one auction and one bid lifecycle at a time
- helper programs handle record splitting, credential setup, and optional batched payouts

Why:

- batch transitions are expensive
- batch size is constrained by Aleo limits
- auctions need correctness first; batching is an optimization layer

### 3. Use Utility Programs As Sidecars

Recommended sidecar programs for `v2.23`:

- `shadowbid_usdcx_record_utils_v1.aleo`
  purpose: split private USDCx token records into settlement-ready denominations
- `shadowbid_usdcx_cred_utils_v1.aleo`
  purpose: batch credential provisioning when many executor-owned records are needed
- `shadowbid_usdcx_batch_settlement_v1.aleo`
  purpose: optional future optimization for batched refund or payout operations

These should support the auction system, not replace it.

### 4. Benchmark ShadowBid Workloads Before Locking The ABI

Before freezing a `v2.23` ABI, run benchmark scenarios closer to actual ShadowBid behavior:

- one auction with 2 to 8 bidders
- refund-heavy cancellation path
- one-winner settlement path
- seller payout plus platform fee payout
- multiple auctions settling concurrently

The benchmark repo is a good pattern for how to measure this.

## Proposed Architecture

### Core Contract

Recommended program target:

- `shadowbid_marketplace_v2_23_usdcx_private.aleo`

The narrower name is intentional. It avoids implying that `ALEO` and `USAD` are equally private on day one.

Core responsibilities:

- auction metadata and state machine
- commitment storage
- reveal verification
- winner selection
- reserve checks
- dispute transitions
- claim ordering for refunds, seller proceeds, and platform fees

### Private Funding Rail

Recommended direction:

- the private bid path should use private `USDCx` records plus credentials
- the bid amount should not be supplied through a public funding transfer
- the commitment should continue to bind `auction_id`, `bidder`, `amount`, and secret randomness

This preserves the good parts of `v2.22` while addressing the remaining amount-leakage problem, but it is currently paused because the repo cannot self-build the required `get_credentials` proof.

### Escrow Model

This is still the hardest open design question.

What `v2.23` must answer explicitly:

- where the privately funded asset lives between commit and settlement
- who can later move it for refund or payout
- how that movement stays compatible with contract-verifiable auction outcomes

The benchmark repo does not answer this directly.

Recommended `v2.23` milestone split:

- `v2.23-alpha`: prove private funding and commitment compatibility
- `v2.23-beta`: prove refund and winner settlement from the chosen private custody model
- `v2.23-final`: market as hidden-amount sealed-bid only after both paths pass end-to-end smoke tests

Current gate before `v2.23-alpha`:

- demonstrate an external proof source for `get_credentials`

Supporting working docs for this milestone split:

- `docs/contracts/v2.23-phase-1-feasibility-spike.md`
- `docs/contracts/v2.23-custody-decision.md`

## Recommended Delivery Phases

### Phase 1: Feasibility Spike

- build a minimal private USDCx bid funding prototype
- confirm credential flow, record flow, and proving costs
- benchmark single-bid commit plus later refund

### Phase 2: Utility Programs

- add record splitting helper
- add credential batching helper
- add benchmark scripts for ShadowBid-specific settlement scenarios

### Phase 3: Core Private Auction Path

- implement private USDCx bid commit path
- keep `BHP256` commitment verification
- keep public mappings free of per-bid amounts
- preserve `v2.22` state machine semantics where possible

### Phase 4: Settlement And Refund Proof

- prove cancellation path with private refund
- prove winner path with private or privacy-preserving payout sequence
- verify reserve, dispute, and payout ordering still behave correctly

### Phase 5: UI And Product Guardrails

- show privacy mode per currency, not as a blanket claim
- disable unsupported private paths in the UI
- rewrite product copy to match the actual deployed behavior

## What Should Not Be In V2.23 MVP

To keep scope realistic, the first private release should not require all of the following at once:

- full private parity for `ALEO`, `USDCx`, and `USAD`
- batch-everything settlement by default
- autonomous delegated reveal redesign
- a brand-new dispute model

Those are all valid future directions, but they should not block the first honest private funding release.

## Acceptance Criteria For V2.23

`v2.23` should only be presented as the hidden-amount sealed-bid upgrade if all of the following are true:

- bid funding in private mode does not expose the amount through a public transfer path
- public mappings still do not expose per-bid committed amounts
- commitment and reveal verification remain cryptographic and contract-derived
- refund path works end to end for private-mode bids
- winner settlement path works end to end for private-mode bids
- UI copy clearly distinguishes private and non-private currency modes

## Recommended Next Step

The best next engineering step is not "write the full contract immediately".

It is:

1. build a `USDCx`-only private funding spike
2. add record-splitting and credential-setup helpers
3. benchmark commit, refund, and settlement paths with ShadowBid-shaped workloads
4. only then freeze the `v2.23` auction ABI

For the concrete execution breakdown, see `v2.23-implementation-checklist.md`.

For the first execution phase before ABI freeze, see `v2.23-phase-1-feasibility-spike.md`.
