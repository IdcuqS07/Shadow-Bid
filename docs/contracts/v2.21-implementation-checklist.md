# ShadowBid v2.21 Implementation And Deployment Checklist

## Purpose

This checklist translates the `v2.21` technical spec into concrete repo work before deploying `shadowbid_marketplace_v2_21.aleo`.

Primary reference:

- `docs/contracts/v2.21-technical-spec.md`

## Recommended Rollout Order

1. update the Leo contract and ABI shape
2. update frontend parsing and transaction helpers
3. update ops/executor logic
4. update env defaults and documentation
5. run testnet validation on the full lifecycle
6. deploy `v2.21`
7. cut frontend traffic to the new program ID

### A4. Transition signature changes

File:

- `contracts/src/main.leo`

Checklist:

- [ ] change `create_auction` inputs to `(auction_id, min_bid_amount, reserve_price, currency_type, asset_type, end_time, reveal_period, dispute_period)`
- [ ] keep `close_auction` signature unchanged unless there is a strong reason to alter it
- [ ] keep `reveal_bid` signature unchanged unless there is a strong reason to alter it
- [ ] add new `settle_after_reveal_timeout(auction_id, settled_at)`

### A5. Transition logic changes

File:

- `contracts/src/main.leo`

Checklist:

- [ ] `create_auction` initializes `reveal_deadline = 0` and `dispute_deadline = 0`
- [ ] `close_auction` writes `reveal_deadline = closed_at + reveal_period`
- [ ] `reveal_bid` validates against `reveal_deadline`
- [ ] `reveal_bid` writes `revealed_at`
- [ ] `settle_after_reveal_timeout` transitions:
  - `CLOSED -> CHALLENGE` if winner exists and reserve is met
  - `CLOSED -> CANCELLED` if no valid reveal exists
  - `CLOSED -> CANCELLED` if reserve is not met
- [ ] `settle_after_reveal_timeout` writes `dispute_deadline = settled_at + dispute_period` when it moves to `CHALLENGE`
- [ ] `finalize_winner` validates against `dispute_deadline`
- [ ] dispute open/resolve flows still work with the new deadline fields

### A6. Deprecations

File:

- `contracts/src/main.leo`

Checklist:

- [ ] remove `determine_winner` from the active flow
- [ ] remove `cancel_auction_reserve_not_met` from the active flow
- [ ] decide one of these approaches before build:
  - fully delete deprecated transitions
  - keep them as wrappers that fail with clear assertions

Recommended choice:

- fully delete them in `v2.21` unless backward wallet compatibility requires preserving the symbol names

### A7. Contract invariants to verify in code review

File:

- `contracts/src/main.leo`

Checklist:

- [ ] no path can move to `CHALLENGE` before `reveal_deadline`
- [ ] no path can finalize before `dispute_deadline`
- [ ] no path can reveal after timeout settlement already moved state away from `CLOSED`
- [ ] no path can leave an auction permanently stuck in `CLOSED`
- [ ] refunds still work for both `SETTLED` losers and timeout-driven `CANCELLED` auctions
- [ ] reserve-miss auctions no longer require seller cleanup

### A8. Build gate

Files:

- `contracts/src/main.leo`
- `contracts/program.json`

Checklist:

- [ ] `leo build` succeeds
- [ ] generated ABI reflects the new fields and new transition names
- [ ] contract size still fits deployment limits with margin

Checklist:

- [ ] contract build is green
- [ ] manual testnet scenario matrix is green

- [ ] ops executor logic is updated before enabling automation
- [ ] deployment transaction hash is recorded
- [ ] post-deploy explorer reads confirm:
  - program exists
  - mappings are queryable
  - at least one new `v2.21` auction can be created and read back

## Go / No-Go Criteria

Do not deploy `v2.21` if any of these are still true:

- `determine_winner` is still referenced in the main UI flow
- executor still recommends `cancel_auction_reserve_not_met`
- frontend still parses only `challenge_end_time`
- any important path still defaults to `shadowbid_marketplace_v2_20.aleo`
- local cache logic cannot distinguish `v2.20` and `v2.21` auction behavior
